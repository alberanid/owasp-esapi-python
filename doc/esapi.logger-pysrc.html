 23      """ 
 24      The Logger interface defines a set of methods that can be used to log 
 25      security events. It supports a hierarchy of logging levels which can be  
 26      configured at runtime to determine the severity of events that are logged, and 
 27      those below the current threshold that are discarded. Implementors should use  
 28      a well established logging library as it is quite difficult to create a  
 29      high-performance logger. 
 30      <P> 
 31      <img src="doc-files/Logger.jpg"> 
 32      <P> 
 33   
 34      The logging levels defined by this interface (in descending order) are: 
 35      <ul> 
 36      <li>fatal (highest value)</li> 
 37      <li>error</li> 
 38      <li>warning</li> 
 39      <li>info</li> 
 40      <li>debug</li> 
 41      <li>trace (lowest value)</li> 
 42      </ul> 
 43   
 44      ESAPI also allows for the definition of the type of log event that is being  
 45      generated. The Logger interface predefines 4 types of Log events:  
 46      SECURITY_SUCCESS, SECURITY_FAILURE, EVENT_SUCCESS, EVENT_FAILURE.  
 47      Your implementation can extend or change this list if desired.  
 48   
 49      This Logger allows callers to determine which logging levels are enabled, and  
 50      to submit events at different severity levels.<br> 
 51      <br>Implementors of this interface should: 
 52   
 53      <ol> 
 54      <li>provide a mechanism for setting the logging level threshold that is  
 55      currently enabled. This usually works by logging all events at and above that  
 56      severity level, and discarding all events below that level. This is usually  
 57      done via configuration, but can also be made accessible programmatically.</li> 
 58      <li>ensure that dangerous HTML characters are encoded before they are logged  
 59      to defend against malicious injection into logs that might be viewed in an  
 60      HTML based log viewer.</li><li>encode any CRLF characters included in log data  
 61      in order to prevent log injection attacks.</li><li>avoid logging the user's  
 62      session ID. Rather, they should log something equivalent like a generated  
 63      logging session ID, or a hashed value of the session ID so they can track  
 64      session specific events without risking the exposure of a live session's ID. 
 65      </li><li>record the following information with each event:</li> 
 66        <ol type="a"> 
 67        <li>identity of the user that caused the event,</li> 
 68        <li>a description of the event (supplied by the caller),</li> 
 69        <li>whether the event succeeded or failed (indicated by the caller),</li> 
 70        <li>severity level of the event (indicated by the caller),</li> 
 71        <li>that this is a security relevant event (indicated by the caller),</li> 
 72        <li>hostname or IP where the event occurred (and ideally the user's source IP 
 73        as well),</li> 
 74        <li>a time stamp</li> 
 75        </ol> 
 76      </ol> 
 77        
 78      Custom logger implementations might also: 
 79      <ol start="6"> 
 80      <li>filter out any sensitive data specific to the current application or  
 81      organization, such as credit cards, social security numbers, etc.</li> 
 82      </ol> 
 83   
 84      In the default implementation, this interface is implemented by PythonLogger,  
 85      which is an inner class in PythonLogFactory.java. PythonLogger uses the logging  
 86      package as the basis for its logging implementation. This default  
 87      implementation implements requirements #1 thru #5 above.<br><br> 
 88      Customization: It is expected that most organizations will implement their own  
 89      custom Logger class in order to integrate ESAPI logging with their logging  
 90      infrastructure. The ESAPI Reference Implementation is intended to provide a  
 91      simple functional example of an implementation. 
 92      """
