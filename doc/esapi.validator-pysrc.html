 21      """ 
 22      The Validator interface defines a set of methods for canonicalizing and 
 23      validating untrusted input. Implementors should feel free to extend this 
 24      interface to accommodate their own data formats. Methods prefixed with "is" 
 25      should return boolean values. Methods with a "get" prefix should return 
 26      valid input or raise an exception. 
 27       
 28      For "get" methods, invalid input should generate a descriptive  
 29      ValidationException, and input that is clearly an attack should generate a 
 30      descriptive IntrusionException. 
 31       
 32      "assert" and "get" methods should accept an optional error_list parameter  
 33      to collect any thrown errors instead of raising them. This error_list  
 34      parameter can accept an instance of the ValidationErrorList class. If the 
 35      errors_list is present, any exceptions are added to the list instead of  
 36      being thrown, and the method returns None. 
 37       
 38       
 39      Implementations must adopt a "whitelist" approach to validation where a 
 40      specific pattern or character set is matched. "Blacklist" approaches that 
 41      attempt to identify the invalid or disallowed characters are much more likely 
 42      to be fooled by encoding or other tricks. 
 43   
 44      @author: Craig Younkins (craig.younkins@owasp.org) 
 45      """
