 24      """
 
 25      The AccessController interface defines a set of methods that can be used in
 
 26      a wide variety of applications to enforce access control. In most 
 
 27      applications, access control must be performed in multiple different
 
 28      locations across the various application layers. This class provides
 
 29      access control for URLs, business functions, data, services, and files.
 
 30      
 
 31      The implementation of this interface will need to access the current User
 
 32      object (from Authenticator.current_user) to determine roles or permissions.
 
 33      In addition, the implementation will also need information about the 
 
 34      resources that are being accessed. Using the user information and the
 
 35      resource information, the implementation should return an access control
 
 36      decision.
 
 37      
 
 38      Implementors are encouraged to implement the ESAPI access control rules,
 
 39      like assert_authorized() using existing access control mechanisms, such as 
 
 40      is_user_in_role() or has_privilege(). While powerful, methods like
 
 41      is_user_in_role() can be confusing for developers, as users may be in
 
 42      multiple roles or possess multiple overlapping privileges. Direct use of
 
 43      these finer grained access control methods encourages the use of complex
 
 44      boolean tests throughout the code, which can easily lead to developer
 
 45      mistakes.
 
 46      
 
 47      The point of the ESAPI access control interface is to centralize access
 
 48      control logic behind easy to use calls like assert_authorized_for_data() so
 
 49      that access control is easy to use and easy to verify. Here is an example 
 
 50      of a very straightforward to implement, understand, and verify ESAPI access
 
 51      control check:
 
 52      
 
 53      try:
 
 54          ESAPI.access_controller().assert_authorized_for_function(
 
 55              "businessFunction" )
 
 56          # execute businessFunction
 
 57      except AccessControlException, extra:
 
 58          # attack in progress
 
 59          
 
 60      Note that in the user interface layer, acccess control checks can be used
 
 61      to control whether particular controls are rendered or not. These checks
 
 62      are supposed to fail when an unauthorized user is logged in, and do not
 
 63      represent attacks. Remember that regardless of how the user interface
 
 64      appears, an attacker can attempt to invoke any business function or access
 
 65      any data in your application. Therefore, access control checks in the user
 
 66      interface should be repeated in both the business logic and data layers.
 
 67      """
