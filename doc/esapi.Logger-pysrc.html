 17  """ 
 18  The Logger interface defines a set of methods that can be used to log 
 19  security events. It supports a hierarchy of logging levels which can be configured at runtime to determine 
 20  the severity of events that are logged, and those below the current threshold that are discarded. 
 21  Implementors should use a well established logging library 
 22  as it is quite difficult to create a high-performance logger. 
 23  <P> 
 24  <img src="doc-files/Logger.jpg"> 
 25  <P> 
 26   
 27  The logging levels defined by this interface (in descending order) are: 
 28  <ul> 
 29  <li>fatal (highest value)</li> 
 30  <li>error</li> 
 31  <li>warning</li> 
 32  <li>info</li> 
 33  <li>debug</li> 
 34  <li>trace (lowest value)</li> 
 35  </ul> 
 36   
 37  ESAPI also allows for the definition of the type of log event that is being generated. The Logger interface   
 38  predefines 4 types of Log events: SECURITY_SUCCESS, SECURITY_FAILURE, EVENT_SUCCESS, EVENT_FAILURE.  
 39  Your implementation can extend or change this list if desired.  
 40   
 41  This Logger allows callers to determine which logging levels are enabled, and to submit events  
 42  at different severity levels.<br> 
 43  <br>Implementors of this interface should: 
 44   
 45  <ol> 
 46  <li>provide a mechanism for setting the logging level threshold that is currently enabled. This usually works by logging all  
 47  events at and above that severity level, and discarding all events below that level. 
 48  This is usually done via configuration, but can also be made accessible programmatically.</li> 
 49  <li>ensure that dangerous HTML characters are encoded before they are logged to defend against malicious injection into logs  
 50  that might be viewed in an HTML based log viewer.</li> 
 51  <li>encode any CRLF characters included in log data in order to prevent log injection attacks.</li> 
 52  <li>avoid logging the user's session ID. Rather, they should log something equivalent like a  
 53  generated logging session ID, or a hashed value of the session ID so they can track session specific  
 54  events without risking the exposure of a live session's ID.</li>  
 55  <li>record the following information with each event:</li> 
 56    <ol type="a"> 
 57    <li>identity of the user that caused the event,</li> 
 58    <li>a description of the event (supplied by the caller),</li> 
 59    <li>whether the event succeeded or failed (indicated by the caller),</li> 
 60    <li>severity level of the event (indicated by the caller),</li> 
 61    <li>that this is a security relevant event (indicated by the caller),</li> 
 62    <li>hostname or IP where the event occurred (and ideally the user's source IP as well),</li> 
 63    <li>a time stamp</li> 
 64    </ol> 
 65  </ol> 
 66    
 67  Custom logger implementations might also: 
 68  <ol start="6"> 
 69  <li>filter out any sensitive data specific to the current application or organization, such as credit cards,  
 70  social security numbers, etc.</li> 
 71  </ol> 
 72   
 73  In the default implementation, this interface is implemented by PythonLogger, which is an inner class  
 74  in PythonLogFactory.java. PythonLogger uses the logging package as the basis for its logging  
 75  implementation. This default implementation implements requirements #1 thru #5 above.<br> 
 76  <br> 
 77  Customization: It is expected that most organizations will implement their own custom Logger class in  
 78  order to integrate ESAPI logging with their logging infrastructure. The ESAPI Reference Implementation  
 79  is intended to provide a simple functional example of an implementation. 
 80  """
